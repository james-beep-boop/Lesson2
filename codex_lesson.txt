MVP scope

1. Primary goal
Build a lesson-plan repository for teachers and administrators with:
- authentication
- class-based permissions
- lesson-plan family/version management
- markdown viewing and editing
- semantic versioning
- comparison between versions
- favorites
- simple in-app messaging with unread alerts
- core admin workflows

2. Explicitly defer for later
Do not implement these beyond minimal placeholders if necessary:
- DOC / DOCX export
- advanced PDF export
- advanced print formatting
- AI-assisted editing or suggestions
- network/device transfer workflows
- rich analytics dashboards
- complex charting
- threaded messaging
- advanced mobile polish beyond responsive usability
- email-change workflow unless it comes almost free from existing auth scaffolding

3. Product rules

Users and access:
- New signups become basic Teachers only.
- All signed-in users can view all lesson plans across all classes.
- A class is exactly subject + grade.
- A Site Administrator assigns users to one or more classes.
- Editors are class-scoped.
- Class Administrators are class-scoped.
- Site Administrators are global.
- At most one Class Administrator may exist per class.
- “Contributor” is not a role. It is simply the user who created a lesson-plan version.

Permissions:
- Teacher:
  - view lesson plans
  - compare versions
  - favorite / unfavorite lesson plans
  - use inbox / messaging
- Editor:
  - all Teacher permissions
  - create new versions by editing lesson plans for assigned classes
- Class Administrator:
  - all Editor permissions for assigned classes
  - mark a version official for assigned classes
  - promote/demote Teacher <-> Editor within assigned classes
  - request deletion of lesson-plan versions in assigned classes
- Site Administrator:
  - full global control
  - manage users
  - assign classes
  - assign class administrators
  - directly delete lesson-plan versions
  - basic admin summary counts

Lesson-plan rules:
- A lesson-plan family is defined by subject + grade + day + language.
- Versions within a family are immutable once saved.
- Editing always creates a new version, never overwrites an existing one.
- Version numbers use semantic format x.y.z only.
- Default bump when saving edits is Patch.
- User may choose Patch, Minor, or Major.
- The system computes the next valid version number; the user does not type arbitrary version strings.
- Version numbers must be unique within a family.
- Only one version in a family may be official at a time.
- Setting a version to official must atomically unset the previous official version in that family.

Deletion rules:
- Only Site Administrators may hard-delete directly.
- Class Administrators may request deletion.
- A deletion request creates an in-app message to the contributor and relevant admins.
- Editors do not directly delete.

Messaging:
- Simple direct inbox-style messaging only.
- Unread message count appears in the top navigation.
- No threading required for MVP.

Important modeling rules:
- Store lesson-plan markdown content in the database for MVP as the canonical source.
- Do not rely on filesystem filenames as the primary identity mechanism.
- Use stable IDs and slugs where useful.

- Enforce “only one class admin per class.”
- Enforce “only one official version per family.”
- Use transactions / service-layer logic for critical business rules, not only UI checks.
